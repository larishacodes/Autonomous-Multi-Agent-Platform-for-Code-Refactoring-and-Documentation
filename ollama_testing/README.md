# Ollama Testing Module
## Autonomous Multi-Agent Platform — Comparative Evaluation

Tests zero-shot Ollama models on the **same prompts** as the main CodeT5+ pipeline.
Produces a side-by-side comparison report for research and paper writing.

> This module does NOT modify the main project in any way.
> It reads prompts generated by the main pipeline's prompting engine and routes them through Ollama.

## Purpose

The main pipeline uses fine-tuned CodeT5+ 770M for refactoring and documentation.
This module answers the research question:

> How does a zero-shot large language model compare to a fine-tuned smaller model
> on Java code refactoring and Javadoc generation?

This comparison is a direct contribution to the research paper.

---

## Architecture

```
Main Pipeline Output                 Ollama Testing Module
outputs/run_TIMESTAMP/
  PROMPT_refactor_agent.txt  ---->  ollama_refactor.py  ---->  codellama:7b
                                                         ---->  deepseek-coder:6.7b
                                                         ---->  gemma3:12b
                                                         ---->  qwen2.5vl:7b

  PROMPT_doc_agent.txt       ---->  ollama_doc.py        ---->  codellama:7b
                                                         ---->  deepseek-coder:6.7b
                                                         ---->  gemma3:12b
                                                         ---->  qwen2.5vl:7b
                                         |
                                  ollama_evaluator.py   (same metrics as main pipeline)
                                         |
                                  ollama_compare.py     (side-by-side report)
```

The split point is after the prompting engine. Parser, smell detector, and prompt
generation are shared. Only the agent (model call) is different.

## Setup

### Step 1 - Verify Ollama is running

Open PowerShell and run:
```
curl http://localhost:11434/api/tags
```

If you see JSON output, Ollama is running.
If you see "Connection refused", run: ollama serve

Note: "bind: Only one usage of each socket address" means Ollama is already running. That is fine.

### Step 2 - Pull models

```
ollama pull codellama:7b
ollama pull deepseek-coder:6.7b
ollama pull gemma3:12b
ollama pull qwen2.5vl:7b

ollama list
```

### Step 3 - Install Python dependencies

```
cd Major_Project_Java
venv\Scripts\activate
pip install requests javalang
```

### Step 4 - Place this module

```
Major_Project_Java/
    main.py
    pipeline.py
    ...
    ollama_testing/          <-- this folder goes here
        run_ollama_test.py
        ollama_client.py
        ollama_refactor.py
        ollama_doc.py
        ollama_evaluator.py
        ollama_compare.py
        README.md
```

## How to Run

### Option A - Full test on all models (recommended)

```
# Step 1: Run main pipeline first to generate prompts
cd Major_Project_Java
python main.py --file SimpleTest.java --mode both

# Step 2: Run Ollama test on that pipeline output
cd ollama_testing
python run_ollama_test.py --pipeline_run ..\outputs\run_20260319_140941

# Step 3: Generate comparison report
python ollama_compare.py --pipeline_run ..\outputs\run_20260319_140941 --ollama_run ollama_outputs
```

### Option B - Test specific models only

```
python run_ollama_test.py --pipeline_run ..\outputs\run_20260319_140941 --models codellama:7b gemma3:12b
```

### Option C - Test on a different Java file

```
cd Major_Project_Java
python main.py --file OrderService.java --mode both

cd ollama_testing
python run_ollama_test.py --pipeline_run ..\outputs\run_NEW_TIMESTAMP
```

### Option D - Refactoring only

```
python ollama_refactor.py --prompt_file ..\outputs\run_TIMESTAMP\PROMPT_refactor_agent.txt --output_dir ollama_outputs
```

### Option E - Documentation only

```
python ollama_doc.py --prompt_file ..\outputs\run_TIMESTAMP\PROMPT_doc_agent.txt --output_dir ollama_outputs
```

## Output Structure

Each model gets its own subfolder. All files are UTF-8 encoded.

```
ollama_outputs/
    codellama_7b/
        refactored_code.java      Refactored Java output
        documentation.md          Generated Javadoc
        eval_refactor.json        Refactoring evaluation scores
        eval_doc.json             Documentation evaluation scores
        result_refactor.json      Raw model response and timing
        result_doc.json           Raw model response and timing

    deepseek-coder_6_7b/
        (same files)

    gemma3_12b/
        (same files)

    qwen2_5vl_7b/
        (same files)

    ollama_summary_TIMESTAMP.json    All scores combined
    comparison_report.txt            Side-by-side vs CodeT5+
```

## Metrics

### Refactoring (eval_refactor.json)

| Metric | Description |
|--------|-------------|
| ast_validity.valid | Parses as valid Java via javalang |
| style_metrics.score | 6 checks: camel case, braces, semicolons, no tabs, no trailing whitespace, no integer discounts |
| logic_correctness.score | 5 checks: uses .equals() not ==, decimal discount values, balanced braces, has return, preserves method name |
| extract_method.applied | Did the model extract helper methods? |
| improvement.loc_delta | Lines of code reduced |
| confidence.score | Weighted combination 0.0 to 1.0 |

Confidence weights:
```
AST validity     25%
Style score      20%
Logic correct    30%
LOC improvement  15%
Extract Method   10%
```

Note: logic_correctness is an additional check not in the main pipeline.
It was added because CodeT5+ output had logic bugs that the original evaluator did not catch.

### Documentation (eval_doc.json)

| Metric | Description |
|--------|-------------|
| completeness.checks.has_description | Has method description |
| completeness.checks.has_param_tags | Has at least one @param |
| completeness.checks.has_return_tag | Has @return tag |
| completeness.checks.has_all_params | All parameters documented |
| completeness.checks.proper_format | Starts with /** ends with */ |
| completeness.checks.no_html_noise | No HTML tags like p or /p |
| parameter_coverage.score | Fraction of params covered 0.0 to 1.0 |
| parameter_coverage.missing | List of undocumented parameters |
| confidence.score | Weighted combination 0.0 to 1.0 |

Confidence weights:
```
Completeness score  35%
Parameter coverage  35%
Has @return tag     20%
Length ok           10%
```

## Comparison Report

Run after all models are tested:

```
python ollama_compare.py \
    --pipeline_run ..\outputs\run_20260319_140941 \
    --ollama_run   ollama_outputs \
    --output       ollama_outputs\comparison_report.txt
```

The report shows:
- Refactoring scores for all models side by side
- Documentation scores for all models side by side
- Visual confidence bars
- Summary verdict with key finding

## Results (codellama:7b vs CodeT5+ on SimpleTest.java)

| Metric | CodeT5+ fine-tuned | codellama:7b zero-shot |
|--------|-------------------|----------------------|
| Refactor AST valid | YES | YES |
| Refactor logic correct | 0.400 (broken) | 1.000 |
| Refactor confidence | 0.791 | 0.830 |
| Extract Method | NO | YES |
| Doc param coverage | 0.000 (none) | 1.000 (all 6) |
| Doc has @return | NO | YES |
| Doc confidence | 0.930 (structural) | 1.000 (real) |

Key finding: Zero-shot CodeLlama 7B outperforms fine-tuned CodeT5+ 770M on both
refactoring logic correctness and documentation completeness.

## Troubleshooting

| Problem | Solution |
|---------|----------|
| Connection refused | Run ollama serve in a separate terminal |
| bind: Only one usage of each socket address | Ollama is already running, ignore this |
| Model not pulled | Run ollama pull model-name |
| UnicodeEncodeError | Fixed in current version, all writes use utf-8 |
| Timeout after 180s | Model slow on CPU, try smaller model or increase timeout in ollama_client.py |
| No Ollama columns in report | Run run_ollama_test.py first to generate per-model folders |
| PROMPT file not found | Run python main.py --file YourFile.java --mode both first |
